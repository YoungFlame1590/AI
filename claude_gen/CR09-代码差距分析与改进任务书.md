## 如何使用本文档

本文档按**任务**组织，每个任务包含：现状（代码已经是什么样）、目标（CR要求做成什么样）、涉及文件、具体改动点、验收标准。Codex 应逐个任务实施，**每完成一个任务就应该能独立编译通过并跑通对应的验收标准**，不需要等全部任务做完再联调。

任务按优先级从高到低排列。任务1和任务2是CR-01（重大变更）的核心缺口，建议优先做。

## 参考实现：design-editor-prototype.html

本次改动附带一份独立原型文件 `design-editor-prototype.html`（基于 Fabric.js 的所见即所得编辑器，纯前端、不依赖后端即可运行）。**任务1不需要从零设计交互方案，应该以这份原型为基础改造、移植进 printshop-v2**，具体对应关系：

| 原型里已有的实现 | 在 printshop-v2 里的用法 |
|---|---|
| Fabric.js 画布初始化、模板尺寸→像素换算、出血/安全区参考线 | 直接复用，`TEMPLATES` 映射表要跟 `DesignTemplate.sizeName` 的真实取值对齐（原型里是硬编码的6种模板，接入后端后应该从 `GET /api/design-templates` 返回的数据里读 `sizeName` 并映射到真实mm尺寸） |
| 添加文字 / 上传图片 / 添加Logo / 生成二维码（qrcodejs） | 交互逻辑原样保留，这就是任务1要求的四个核心操作 |
| 右侧属性面板（字号/颜色/加粗/对齐/不透明度/替换图片） | 原样保留 |
| `versions` 数组 + `saveVersionBtn` + 版本缩略图列表 | **需要替换**：原型里版本是存在浏览器内存里的（`let versions = []`），移植时要把 `saveVersionBtn` 的点击事件改成调用真实接口 `POST /api/design-projects/{id}/versions`，历史版本列表改成从 `GET /api/design-projects/{id}` 返回的 `versions` 字段渲染，点击某个版本调用 `POST /api/design-projects/{id}/restore/{versionNo}` 后再 `loadFromJSON` |
| `orderBtn` 一键下单（原型里只是 `console.log` + `alert` 模拟） | **需要替换**：改成调用真实接口 `POST /api/design-projects/{id}/submit-order`，并在下单面板里加上纸张/工艺下拉选择（配合任务2的后端字段） |
| 模板尺寸下拉 `templateSelect`（原型里6个选项是硬编码的） | **需要替换**：改成根据当前设计项目关联的 `DesignTemplate` 自动确定尺寸并锁定（CR原文写的是"尺寸由模板自动确定"，客户不应该在下单/设计阶段随便切换模板尺寸） |

落地方式建议：把 `design-editor-prototype.html` 的 `<style>` 和 `<script>` 部分拆分整理进任务1里提到的 `static/js/design-editor.js`（逻辑）和 `static/styles.css`（样式，或独立的 `design-editor.css`），HTML结构部分整合进 `renderDesignProjectDetail` 生成的DOM里，而不是保留成一个独立的 `.html` 页面——这样才能吃到现有的登录态、路由、`js/api.js` 里封装好的请求方法（带鉴权头等），避免重复造轮子或者出现两套鉴权逻辑。

## 任务1：所见即所得编辑器前端重构（CR-01 核心缺口）

### 现状

`src/main/resources/static/js/render.js` 的 `renderDesignProjectDetail` / `drawDesignCanvas` 把"在线设计"做成了：一个只读 `<canvas>` 预览 + 一个 `<textarea name="canvasJson">` 文本框。客户要"设计"，实际是手写 JSON 字符串塞进文本框，`drawDesignCanvas` 只是把 `JSON.parse(canvasJson).objects` 数组画成静态预览，**没有任何交互**：不能点击添加文字、不能上传/替换图片、不能生成二维码、不能拖拽或调整元素位置和大小。

对应文件：
- `src/main/resources/static/js/render.js`（`renderDesignProjectDetail`、`drawDesignCanvas`）
- `src/main/resources/static/js/main.js`（`buildDesignProjectBody`、`buildDesignVersionBody`、`buildDesignSubmitOrderBody`）
- `src/main/resources/static/js/config.js`（`designTemplates`、`designProjects` 模块配置）
- `src/main/resources/static/index.html`

### 目标

把"在线设计"详情页替换成真正的所见即所得编辑器，功能对齐CR-01原文的四点：
1. 文字排版：点击添加文字，双击/选中后可编辑内容、字号、颜色、加粗、对齐
2. 图片替换：上传图片、替换已有图片
3. 二维码生成：输入内容，生成二维码并作为元素插入画布
4. Logo添加：上传Logo图片，插入画布（默认缩小定位到角落）

同时保留（复用）已有后端能力：
- 版本保存 → 调 `POST /api/design-projects/{id}/versions`
- 历史版本回溯 → 调 `POST /api/design-projects/{id}/restore/{versionNo}`
- 提交订单 → 调 `POST /api/design-projects/{id}/submit-order`

### 具体改动点

1. **渲染引擎**：改用 Canvas 库（推荐 `fabric.js`，通过 `<script src="https://cdn.jsdelivr.net/npm/fabric@5.3.0/dist/fabric.min.js"></script>` 或本地 vendor 文件引入，注意生产环境应把库文件下载到 `static/vendor/` 下自托管，不要依赖外部CDN可用性）。**本仓库根目录随附的 `design-editor-prototype.html` 已经用 Fabric.js 实现了这一整套交互，请以它为基础改造，不要重新设计交互方案**，具体移植对照见上方"参考实现"一节。

2. **新增独立模块文件** `static/js/design-editor.js`，不要把交互逻辑塞进通用的 `render.js`（那个文件是给其它模块的通用表格/表单渲染用的，设计编辑器的复杂交互应该独立管理，避免污染通用渲染逻辑）。该文件需要暴露一个入口函数，例如：
   ```js
   export function mountDesignEditor(containerEl, project, template, handlers) { ... }
   ```
   `handlers` 包含 `onSaveVersion(canvasJson)`、`onRestoreVersion(versionNo)`、`onSubmitOrder(payload)` 三个回调，内部去调已有的 API 封装（`js/api.js`）。

3. **画布尺寸应该跟随模板规格**：`DesignTemplate.sizeName` 目前是字符串（如 "A4"、"90x54mm"），建议在前端维护一份 `sizeName -> {widthMm, heightMm}` 的映射表（可以从模板管理里的固定几种规格开始：名片90×54、宣传单A5/A4、海报A3、展架、折页、画册封面），用于设置画布真实像素尺寸并显示出血/安全区参考线（这是印刷场景的标志性细节，客户和店员都会关心）。

4. **工具栏功能**（对照现有 `DesignService`/`DesignController` 已有接口即可，不需要改后端）：
   - 添加文字：`new fabric.IText(...)`
   - 上传/替换图片：`<input type=file>` + `FileReader` 转 base64 → `fabric.Image.fromURL`
   - 生成二维码：引入轻量二维码库（如 `qrcodejs`），生成后转成 `fabric.Image` 插入画布
   - 添加Logo：与图片上传逻辑一致，默认缩放到画布宽度的 15%~20%，定位到右下角

5. **保存/加载 canvasJson**：
   - 保存时用 `fabricCanvas.toJSON()` 序列化，作为字符串传给 `saveVersion` 接口的 `canvasJson` 字段
   - 加载已有项目时用 `fabricCanvas.loadFromJSON(project.canvasJson, callback)` 还原
   - 回溯历史版本时，拿到 `version.canvasJson` 后同样走 `loadFromJSON`

6. **`index.html` / `config.js` 调整**：`designProjects` 模块的详情页渲染需要从通用的 record-form 渲染路径里摘出来，改为调用 `mountDesignEditor`（类似现在 `render.js` 里已经对 `designProjects` 模块做了特殊分支 `renderDesignProjectDetail`，保留这个分支入口，只是内部实现换掉）。

### 验收标准

- 客户登录后进入"在线设计"，选择一个已发布模板创建设计项目，能在画布上：添加一段文字并修改字号颜色、上传一张图片、生成一个二维码、上传一个Logo，四个操作全部通过鼠标点击/拖拽完成，全程不需要手写或编辑 JSON 文本
- 点击"保存版本"后，历史版本列表出现新记录；点击某个历史版本缩略图，画布内容正确回溯
- 点击"提交订单"，能看到規格信息被自动带出（配合任务2）

## 任务2：设计→订单规格字段补全（CR-01）

### 现状

`DesignService.submitOrder` 目前只把这些字段传给 `OrderService.createOrder`：`productType`、`colorMode`、`pageCount`、`copies`、`deliveryMode`、`priority`。

`PrintOrder` 实体（`domain/PrintOrder.java`）没有 `sizeName`（尺寸）、`paperType`（纸张）、`craftType`（工艺）字段。`paperType` 目前只存在于生产阶段的 `JobTicket` 实体里，由店员在生产环节事后填写,而不是客户下单时就带出来的。`colorMode` 字段还被复用来表示"颜色/工艺"（`OrderPricingPolicy` 里对应的校验提示语就是"颜色/工艺"），语义上是混在一起的。

这与CR-01原文"在线下单时自动填写印刷规格参数——尺寸由模板自动确定、纸张和工艺可选"不符：客户下单那一刻选不了纸张，尺寸也没有留痕到订单上。

### 目标

1. `PrintOrder` 新增字段：`sizeName`（String，下单时由模板自动带入，不可由客户随意修改）、`paperType`（String，客户下单时可选）、`craftType`（String，客户下单时可选，例如"覆膜"、"烫金"、"过UV"等，与`colorMode`拆分开，语义不再混用）。

2. `DesignTemplate` 已有 `sizeName` 字段，`submitOrder` 时直接把 `template.sizeName` 写入 `PrintOrder.sizeName`，不允许客户在提交订单时覆盖这个值（尺寸由模板确定，这是CR原文明确写的）。

3. `paperType`、`craftType` 在提交订单请求体（`DesignController.submitOrder` 的 `payload`）里作为可选字段传入，如果客户没选，给一个默认值（例如"标准纸"、"无特殊工艺"）。

### 涉及文件

- `domain/PrintOrder.java`：新增三个字段
- `mis/design/DesignService.java`：`submitOrder` 方法里补充字段映射
- `mis/order/OrderService.java`：`createOrder` 方法签名/内部逻辑可能需要接收这几个新字段（视其当前实现而定，如果 `PrintOrder` 是直接传入的请求对象则不需要改方法签名，只需要在 `DesignService.submitOrder` 里把值设置到 request 对象上）
- `resources/db/migration/`：新增迁移脚本（见任务3，可以合并到同一个迁移文件里）
- 前端 `design-editor.js`（任务1产出）：在"提交订单"面板里增加纸张、工艺的下拉选择控件

### 验收标准

- 提交订单后，`print_orders` 表里能看到 `size_name` 字段的值等于所选模板的规格，且客户无法在下单表单里修改这个值
- `paper_type`、`craft_type` 有值写入（客户选了就是所选值，没选就是默认值），且在店员/生产端能看到这两个字段用于安排生产

## 任务3：canvas_json 字段扩容（CR-01）

### 现状

`resources/db/migration/V10__cr09_feature_increment.sql` 里，`design_templates`、`design_projects`、`design_project_versions` 三张表的 `canvas_json` 列都定义为 `VARCHAR(5000)`，对应实体类 `@Column(length = 5000)`。

一旦任务1的编辑器支持插入图片/Logo（通常以 base64 内嵌），哪怕一张压缩过的小图也很容易超过5000字符，会直接写库失败。

### 目标

新增迁移脚本 `V11__design_canvas_capacity.sql`，把三张表的 `canvas_json` 列改为 `TEXT`（或数据库支持的更大类型，如 MySQL 的 `LONGTEXT`），并同步修改对应实体类的 `@Column` 注解（去掉 `length = 5000` 限制，或显式加 `columnDefinition = "LONGTEXT"`）。

### 涉及文件

- 新建 `resources/db/migration/V11__design_canvas_capacity.sql`
- `domain/DesignTemplate.java`、`domain/DesignProject.java`、`domain/DesignProjectVersion.java`：`canvasJson` 字段的 `@Column` 注解调整

### 长期建议（可选，不强制本轮实现）

如果后续图片资源变多，更稳妥的做法是图片本身走文件上传接口存对象存储（复用现有的订单文件上传能力），`canvasJson` 里只存图片的 URL/文件ID引用，而不是内嵌 base64。这样可以避免设计稿JSON随着图片增多无限膨胀。这一条先记录在这里，不作为本轮任务的强制验收项。

### 验收标准

- 保存一个包含至少一张图片（原始文件 300KB 左右）的设计稿版本，不报错，能正常存取

## 任务4：耗材需求预测（CR-02 缺失项）

### 现状

`mis/replenishment/ReplenishmentService.java` 已经实现了动态安全库存（近90天消耗速度 + 供应商提前期）和采购建议（结合批量折扣阶梯算最优采购量），对应CR-02描述的第1、2点。**第3点"采购预测"完全没有实现**：CR原文要求"基于历史同期（去年同月、上月）和门店增长趋势，预测未来30天的耗材需求，生成预采购计划供采购员参考"，代码里没有任何相关字段或方法。

### 目标

新增一个预测方法和接口，输出未来30天的耗材需求预测量，计算逻辑至少覆盖：
- 去年同月消耗量（如果有历史数据）
- 上月消耗量
- 门店层面的增长趋势（例如最近3个月消耗量的环比增长率）

给出一个综合预测值（具体加权公式可以先用简单方案，比如 `预测量 = 上月消耗量 × (1 + 最近3个月平均环比增长率)`，如果没有足够历史数据则退化为"近90天日均消耗 × 30"）。

### 涉及文件

- `mis/replenishment/ReplenishmentService.java`：新增 `forecastNextThirtyDays(String username)` 方法，复用已有的 `InventoryConsumption` 相关repository
- `mis/replenishment/ReplenishmentController.java`：新增 `GET /api/replenishment/forecast` 接口
- 前端：`purchaseSuggestions` 模块页面（`config.js`/`render.js`）增加一个"未来30天预测"的展示区块或独立tab

### 验收标准

- 调用 `GET /api/replenishment/forecast`，能返回每个SKU未来30天的预测需求量，且预测值会随门店消耗趋势变化而变化（不是写死的固定值）
- 前端能看到预测列表，采购员可以参考这份预测和现有的采购建议一起决策

## 任务5：门店服务质量排行看板（CR-03 缺失项）

### 现状

`mis/feedback/FeedbackService.java` 已经实现了评价提交（四维度打分）、差评（≤2星）自动生成客诉工单、30天未消费客户回访提醒，对应CR-03第1、2、4点都做到了。**第3点"门店服务质量排行"没有实现**：CR原文要求"总部看板展示各门店的综合评分趋势、差评率、客诉处理时效等指标"，目前 `reporting`/`dashboard` 模块里没有相关的聚合统计代码。

### 目标

新增一个门店维度的服务质量统计接口，至少包含：
- 各门店综合评分（近30天/近90天可切换，或先做一个固定周期）
- 各门店差评率（评分≤2星的订单占比）
- 各门店客诉处理时效（从工单创建到处理完成的平均耗时，对照CR里"24小时内回复客户"的要求，可以额外标出超时未处理的工单数）

排行结果按综合评分或差评率排序，供总部在看板页面查看。

### 涉及文件

- `mis/reporting/ReportingService.java` / `ReportController.java`：新增门店质量排行的查询方法和接口，例如 `GET /api/reports/store-quality-ranking`
- 需要读取 `ServiceReview`、`ComplaintTicket` 相关repository（`FeedbackService` 里已经在用，可以直接复用对应的 repository 依赖）
- 前端：`dashboard`/`reports` 模块页面新增一个排行表格

### 验收标准

- 总部角色（OPS/ADMIN）登录后能看到一个门店排行列表，包含综合评分、差评率、平均客诉处理时长三列，且数据是从真实的 `ServiceReview`/`ComplaintTicket` 记录聚合计算出来的，不是写死的示例数据

## 任务6：配送对接说明性调整（CR-04，低优先级）

### 现状

`mis/delivery/DeliveryService.java` 的报价、下单、追踪流程都实现了，但渠道名称是用 `"模拟" + channelName` 拼接的字符串，费用是按包裹重量的简单公式估算，**并没有真正调用美团跑腭/顺丰/闪送等第三方API**。作为原型/课程项目阶段这样处理是合理的，不需要在本轮强制对接真实第三方API（涉及资质申请、密钥等，超出当前阶段范围）。

### 目标（可选，视时间/答辩要求决定是否做）

如果只是为了让代码/文档表述更严谨，不产生歧义，建议：
1. 在 `DeliveryService` 里把渠道相关的方法和字段名加上明确注释，说明当前是"模拟适配层"（mock adapter），并预留一个接口（如 `DeliveryChannelAdapter` 接口 + `MockDeliveryChannelAdapter` 实现类），方便未来接入真实API时只需新增一个实现类，不用改动上层调用逻辑
2. 不需要改变现有的业务流程和数据结构

### 验收标准

- 如果选择实施：新增 `DeliveryChannelAdapter` 接口后，现有的报价/下单/追踪流程行为不变（回归测试通过），且代码里能清楚看到"模拟层"和"业务流程层"是分离的
- 如果选择不实施：在答辩文档里注明"CR-04当前为模拟对接，真实第三方API对接列为后续迭代计划"即可，不需要动代码
