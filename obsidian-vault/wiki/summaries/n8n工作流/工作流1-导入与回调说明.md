# 工作流1：导入与回调说明

## 基本信息

- 工作流名称：需求开发全流程（工作流1）
- 项目：图文快印门店连锁管理系统
- n8n 默认地址：`http://localhost:5678`
- agent-app 默认地址：`http://127.0.0.1:8000`
- 工作流文件：`obsidian-vault/wiki/summaries/n8n工作流/需求开发全流程-工作流1.json`

## 导入步骤

1. 启动 n8n。
2. 打开 n8n 页面，进入 Workflows。
3. 选择 Import from File 或 Import from Clipboard。
4. 导入 `需求开发全流程-工作流1.json`。
5. 检查 Webhook 路径、HTTP Request URL 和 Code 节点。
6. 激活工作流。

## 启动工作流

启动前先双击根目录 `启动需求获取页面.bat`，确保 agent-app 已运行。

```bash
curl -X POST http://localhost:5678/webhook/requirements-workflow-1/start \
  -H "Content-Type: application/json" \
  -d '{
    "apiKey": "YOUR_BAILIAN_KEY",
    "agentBaseUrl": "http://127.0.0.1:8000",
    "acceptA5Risks": false,
    "ccbConclusion": "通过（无保留意见）"
```

## A2 风险提示

工作流在 `A2 需求质量分析` 后进入 `A2风险提示` Code 节点。

- A2 结构化结果中没有 `高`、`紧急`、`严重`、`阻塞` 问题时，标记为 `normal` 并继续进入 A3。
- 存在上述问题时，标记为 `high`，输出问题数量、A2 报告路径和风险提示，然后继续进入 A3。
- n8n 不再默认调用 A2 回退接口，避免批量工作流反复补充 notes；如 CCB 或教师要求补访，可在页面手动执行 A2 回退后重新运行工作流。

## CCB 审批回调

当工作流执行到 `记录CCB审批请求` 节点时，agent-app 会在知识库写入待审批文件：

```text
obsidian-vault/wiki/summaries/n8n工作流/ccb-pending/
```

随后工作流停在 `CCB审批等待`。推荐使用根目录脚本提交审批：

```text
一键CCB审批.bat
```

脚本会读取最新 `CCB待审批-*.json` 中的运行时 resume URL，并用 UTF-8 JSON 提交审批结论，不需要手工拼接 `$execution.resumeUrl`。

### 情形1：CCB 审批通过

```json
{
  "decision": "approved",
  "comments": "CCB 审批通过，无保留意见",
  "acceptA5Risks": false
}
```

### 情形2：CCB 带 A5 风险批准

```json
{
  "decision": "approvedWithRisk",
  "comments": "接受 A5 报告中未解决问题，带风险批准进入需求基线",
  "acceptA5Risks": true
}
```

### 情形3：CCB 审批不通过

```json
{
  "decision": "rejected",
  "comments": "当前 SRS 风险不可接受，本轮终止",
  "acceptA5Risks": false
}
```

### 情形4：CCB 要求退回 A4 修改 SRS

```json
{
  "decision": "revise",
  "comments": "请退回 A4 修订 SRS 后重新运行 A5 验证",
  "acceptA5Risks": false
}
```

## agent-app 接口关系

| 阶段 | n8n 节点 | agent-app 接口 |
|---|---|---|
| A1 | A1b 批量访谈并保存 | `POST /api/n8n/a1b-batch` |
| A2 | A2 需求质量分析 | `POST /api/n8n/a2-analyze` |
| A2风险 | A2风险提示 | n8n Code 节点，仅提示不阻断 |
| A2手动回退 | 页面手动回退或自定义节点 | `POST /api/n8n/a2-rollback` |
| A3 | A3 UML 建模 | `POST /api/a3/model` |
| A4 | A4 SRS 生成 | `POST /api/a4/generate` |
| A5 | A5 需求验证 | `POST /api/a5/validate` |
| CCB | 记录CCB审批请求 | `POST /api/n8n/ccb-pending` |
| A6 | A6 创建需求基线 | `POST /api/a6/create-baseline` |

工作流不会调用已删除的自动闭环接口：

- `/api/a5/auto-repair`
- `/api/a3/repair-from-a5`

## Webhook 超时说明

- n8n 默认端口为 `5678`。
- Webhook 的实际 URL 以 n8n 节点面板显示为准，Test URL 只在测试执行时可用，Production URL 需要激活工作流。
- `CCB审批等待` 使用 Wait 节点的 webhook resume 模式。长时间暂停是否过期取决于 n8n 的执行保留和超时配置。
- 不要手工使用字面量 `$execution.resumeUrl`；请使用待审批 JSON 中已经生成好的 `resumeUrl`，或直接运行 `一键CCB审批.bat`。
