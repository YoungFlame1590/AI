# CR09 演示数据补充任务书（耗材预测 / 门店质量排行）

> 面向对象：Codex
> 项目：generated-code/printshop-v2
> 背景：CR-01/CR-02/CR-03/CR-04 的功能代码已经实现（见上一份《CR09-代码差距分析与改进任务书》），但耗材需求预测和门店服务质量排行在实际运行时无法体现效果。本任务书只解决"演示数据"问题，不改动业务逻辑代码。

## 根因说明（请先读，不要凭感觉造数据）

### 预测功能为什么看不出差异

`InventoryService` 里消耗记录的时间戳是写死的：

```java
consumption.consumedAt = com.printshop.mis.shared.MisSupport.now();
```

而 `ReplenishmentService.forecastFor()` 需要区分"上月消耗"、"去年同期消耗"、"近90天日均"三个时间桶来做加权预测。只要消耗记录都是"刚生成的"，上月和去年同期永远是0，算法只能一直走兜底分支（近90天日均×30），预测结果就不会呈现出"环比趋势""同期对比"的效果。**这个问题无法靠多跑几次下单流程解决**，因为正常业务流程走一遍订单只会产生"现在"这个时间点的消耗记录；必须直接往数据库里补一批带真实历史时间戳的消耗记录。

### 排行功能为什么看不出差异

现有种子数据（`V1__print_mis_schema.sql`）里有3家门店（大学城店/市中心店/西区店），但测试账号 `customer`、`clerk`、`manager` 全部绑定 `store_id = 1`（大学城店），门店2、3没有任何可登录账号。`PrintOrder.storeId` 是从下单用户的 `storeId` 带出来的（`OrderService`：`order.storeId = user.storeId`），所以**不管生成多少订单，永远只会落在同一家店**，排行自然无从谈起。另外 `scripts/seed-demo-data.ps1` 里完整搜索过，没有任何一行调用评价提交相关接口，评价数据目前是0条，不是不够多，是完全没有。

## 任务A：补充多门店测试账号

### 现状
只有门店1（大学城店）有 `customer`/`clerk`/`manager` 账号，门店2、3无人可登录下单。

### 目标
新增一份迁移脚本，给门店2（市中心店）、门店3（西区店）各补一套测试账号，用户名规则和现有的保持一致但加门店后缀，避免和 `V1` 里已有账号冲突。

### 涉及文件
新建 `src/main/resources/db/migration/V12__seed_multi_store_accounts.sql`：

```sql
INSERT INTO user_accounts(username, password, role, display_name, store_id) VALUES
('customer_b', 'demo123', 'CUSTOMER', '市中心店客户', 2),
('clerk_b', 'demo123', 'CLERK', '市中心店前台', 2),
('manager_b', 'demo123', 'MANAGER', '市中心店店长', 2),
('customer_c', 'demo123', 'CUSTOMER', '西区店客户', 3),
('clerk_c', 'demo123', 'CLERK', '西区店前台', 3),
('manager_c', 'demo123', 'MANAGER', '西区店店长', 3);
```

（具体密码加密方式需要跟 `V1` 里已有账号的存储方式保持一致，如果密码字段是走应用层加密而不是明文存库，需要用和 `V1`/现有账号一致的方式生成，不要直接明文塞进去导致登录不了。）

### 验收标准
- 用 `customer_b`/`clerk_b` 登录能看到自己属于市中心店，创建的订单 `storeId` 是2；`customer_c`/`clerk_c` 同理对应门店3

## 任务B：补充历史耗材消耗数据（仅demo环境）

### 现状
`inventory_consumptions` 表里只有"当下"产生的记录，没有跨月、跨年的历史数据，`forecastFor()` 的上月/去年同期分支永远命中不了。

### 目标
补一批**跨13个月**的历史消耗数据，覆盖现有4个SKU（`PAPER-A4-80G`、`PAPER-COATED-300G`、`INK-COLOR`、`BINDING-CONSUMABLE`），并且**故意设计出有差异的模式**，这样预测结果才有东西可看：
- `PAPER-A4-80G`：模拟"开学季"效应，每年9月消耗量明显高于其他月份（对应CR-02变更理由里提到的场景），今年整体消耗量比去年同期略有增长
- `INK-COLOR`：稳定小幅增长趋势（每月环比+3%左右），没有明显季节性
- `PAPER-COATED-300G`：基本持平，波动很小
- `BINDING-CONSUMABLE`：轻微下降趋势

每个SKU每月拆成3~5条记录（不同门店、不同订单号），不要一个月只插一条大数——这样 `recent90DailyAverage` 这类日均类指标也能算得更真实。

### 这份数据不应该进生产环境，需要按profile隔离

`application-demo.yml` 目前和生产用同一套migration目录。建议在 `application-demo.yml` 里加一行，让demo profile额外加载一个专属迁移目录：

```yaml
spring:
  flyway:
    locations: classpath:db/migration,classpath:db/migration/demo
```

然后把这份历史消耗数据放到新建的 `src/main/resources/db/migration/demo/V1__seed_consumption_history.sql`（注意这个目录下的版本号是独立的一套，从V1开始即可，不会和主目录的V12冲突），保证正式环境不会被这批构造数据污染。

### 涉及文件
- `src/main/resources/application-demo.yml`：加 `spring.flyway.locations` 配置
- 新建 `src/main/resources/db/migration/demo/V1__seed_consumption_history.sql`

### 验收标准
- 用 `demo` profile 启动后，调用 `GET /api/replenishment/forecast`，`PAPER-A4-80G` 的 `sameMonthLastYearConsumption` 和 `averageMonthlyGrowthRate` 应该是非零的、有意义的数值，四个SKU之间的预测结果应该能看出明显差异（有的涨有的跌有的持平）
- 用非demo profile（正常/生产配置）启动，`inventory_consumptions` 表里不应该出现这批构造数据

## 任务C：扩展种子脚本，覆盖多门店评价数据

### 现状
`scripts/seed-demo-data.ps1` 只用单一 `customer`/`clerk` 账号跑单店订单，且完全没有评价提交步骤。

### 目标
1. 脚本增加门店维度：把现有 `$Orders` 笔订单按比例分给三家门店的账号（比如按 `Orders % 3` 轮流用 `customer`/`customer_b`/`customer_c` 下单），依赖任务A新增的账号
2. 订单流转到 `DONE` 状态后，增加一步调用评价提交接口（需要先看 `FeedbackController` 暴露的具体路径和参数，脚本里目前所有接口调用都是通过 `Invoke-Api` 封装的，照现有模式加一个 `Submit-Review` 函数即可），四个评分维度给出有区分度的随机值：
   - 门店2（市中心店）整体给较高分（4~5星），体现"服务好"
   - 门店3（西区店）里混入几条2星以下的差评（比如每10条里放1~2条），用来触发差评自动工单（`FeedbackService.createComplaint`），这样客诉工单和处理时效这两项指标也有数据可看
   - 门店1（大学城店）给中等偏上评分，作为对照组

### 涉及文件
- `generated-code/printshop-v2/scripts/seed-demo-data.ps1`

### 验收标准
- 脚本跑完后，`GET /api/reports/store-quality-ranking`（或对应报表接口里的 `storeQualityRanking` 字段）能看到三家门店都有数据，且评分、差评率有明显差异，排序结果符合"门店2 > 门店1 > 门店3"这样的预期顺序
- 门店3至少产生1~2个自动生成的客诉工单，可以在客诉工单列表里看到

---

## 执行顺序

任务A必须最先做（任务C依赖它提供的多门店账号）。任务B可以和任务A/C并行，互不依赖。全部完成后，建议顺序是：先跑迁移（任务A、B的SQL），再跑一遍扩展后的种子脚本（任务C），最后分别访问预测接口和排行接口验证效果。
