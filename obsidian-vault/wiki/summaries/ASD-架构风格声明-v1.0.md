# ASD 架构风格声明 (Architecture Style Declaration)

## 1. 架构风格
**分层架构（模块化单体 Modular Monolith）**
系统按业务领域划分为6个核心模块（ORD, QUO, PRO, DLV, FIN, AUD），在单一进程内运行，但通过严格的包结构、依赖倒置和内部接口实现模块间的逻辑隔离。

## 2. 核心原则
1. **领域驱动与高内聚**：严格遵循SRS 3.1的模块划分，每个模块拥有独立的领域模型、应用服务和持久化逻辑（依据UML用例图边界）。
2. **状态机集中管控**：所有核心实体（订单、排产、配送）的状态流转必须通过统一的状态机引擎驱动，严禁在业务代码中硬编码状态变更（依据SRS 1.3 五步闭环定义）。
3. **审计优先与不可篡改**：所有写操作、配置变更、状态流转必须经过审计切面，生成不可篡改的快照日志（依据SRS REQ-AUD-001, REQ-AUD-002）。
4. **配置与代码分离**：所有业务阈值（如折扣矩阵、信用额度、外协占比15%）必须抽象为配置项，支持热更新（依据SRS REQ-QUO-004, REQ-FIN-001, 附录C CCB裁决）。

## 3. 强制约束
1. **禁止跨模块直接数据库访问**：模块间禁止直接Join表或访问对方的Repository，必须通过模块暴露的Application Service接口或领域事件进行通信。
2. **禁止绕过状态机**：任何实体状态变更必须调用 `StateMachine.transit()`，禁止直接 `entity.setStatus()`（依据SRS 2.4 约束1 强制五步闭环）。
3. **禁止硬编码业务规则**：折扣决策矩阵、路由权重、SLA时效等规则必须存储在规则引擎或配置中心，禁止在代码中写死 `if (discount > 0.1)`（依据SRS REQ-QUO-004, REQ-DLV-001）。
4. **财务数据强一致性**：涉及支付、退款、发票、对账的操作，必须在同一个本地数据库事务内完成，禁止使用最终一致性（依据SRS 2.4 约束2 100%线上支付，REQ-FIN-002 日结对账）。

## 4. 推荐实践
1. **依赖倒置原则 (DIP)**：模块间依赖通过定义在消费方的接口（Interface）来实现，提供方实现该接口。
2. **切面编程 (AOP)**：使用AOP实现审计日志（AUD模块）、权限校验、分布式锁，避免业务代码污染。
3. **策略模式与规则引擎**：针对折扣审批路由（REQ-QUO-004）和跨店派单路由（REQ-DLV-001），采用策略模式结合轻量级规则引擎（如Aviator或QLExpress）实现决策矩阵。
4. **防腐层 (ACL)**：在对接第三方支付（REQ-FIN-001）、企微（REQ-AUD-001）、税务发票等外部系统时，必须建立防腐层，隔离外部模型变化。

## 5. 明确禁止项
1. **禁止在Controller层编写业务逻辑**：Controller仅负责参数校验与协议适配，业务逻辑必须下沉至Application Service或Domain Service。
2. **禁止使用分布式事务（如Seata/Saga）**：当前架构为单体，所有业务均在同一数据库实例，严禁引入分布式事务增加不必要的复杂度和性能损耗。
3. **禁止前台手工抹零或 bypass 审批**：系统必须通过代码强制拦截尾差和越权操作，禁止提供“强制通过”的后门（依据SRS REQ-FIN-003, REQ-AUD-002）。
