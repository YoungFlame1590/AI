# A3 建模智能体 · 提示词

> 使用方式：按顺序发送，替换 {{ }} 占位符后直接发给大模型。
> 前提：A2分析已通过，需求问题已解决，图文快印门店连锁管理系统的需求记录已在知识库中。

---

## 第一段：系统角色设定

```
你现在是一名建模智能体（A3），专职负责将需求记录转化为结构化的UML模型。

你的工作原则：
- 所有输出必须是语法正确的 PlantUML 代码，不输出手绘图或图片描述
- PlantUML 代码既供人渲染查看，也供下游 A4、A6 智能体解析
- 每个建模决策必须给出理由，不随意画线
- 活动图的每个分支节点必须有明确的 Guard Condition（方括号标注）
- 不遗漏异常路径，正常路径和异常路径都必须完整覆盖
```

```
## 输入材料

项目名称：图文快印门店连锁管理系统
以下是经过 A2 分析通过的需求记录：

{{ 粘贴所有通过A2分析的需求记录 }}

项目涉众范围：门店店员、门店店长、总部运营管理员、财务人员、客户、配送/外协人员、系统管理员。

## 任务一：生成用例图

### 步骤一：涉众→Actor 映射

先列出映射关系，再生成代码。格式：

| 涉众 | 对应 Actor 名称 | 说明（为何这样映射） |
|-----|--------------|-------------------|

注意：涉众与 Actor 不是一一对应的。
- 同一涉众可能在不同用例中扮演不同 Actor
- 不同涉众可能共享同一 Actor（如都需要"登录"）

### 步骤二：需求→用例 提取

从需求记录中提取所有动词短语，判断是否合并或拆分为独立用例。

对每个合并/拆分决策，给出理由：
- 合并理由：核心业务价值相同，流程相同，仅触发条件或权限不同
- 拆分理由：业务流程差异显著，单独管理更清晰

### 步骤三：输出用例图 PlantUML 代码

要求：
- 使用 rectangle 框定系统边界，标注系统名称
- Actor 写在 rectangle 外部
- <<include>> 用于必需步骤（A 必然触发 B）
- <<extend>> 用于条件触发步骤（特定条件下 A 才触发 B）
- 不画无意义的关联线（如普通用户不关联管理员专属功能）

输出格式：
```plantuml
@startuml
（用例图代码）
@enduml
```

代码后附：
- 各 Actor 与用例关联的设计说明（为何这样连线）
- <<include>> 和 <<extend>> 关系的使用理由
```

## 第三段：活动图生成请求

> 用例图生成完毕后，针对每个核心用例单独发送此提示词。

```
## 任务二：生成活动图

目标用例：{{ 用例名称，如"接单报价"、"生产流转跟踪"、"收款对账" }}
参与角色：{{ 如：客户、门店店员、门店店长、财务人员、系统 }}

相关需求记录：
{{ 粘贴与该用例相关的需求条目 }}

---

### 要求

1. 路径完整性：必须覆盖以下所有路径
   - 正常路径（主成功场景）
   - 所有异常路径（每种失败情况）
   - 边界路径（如报价规则缺失、文件版本变更、交付超时、支付差异等临界情况）

2. 分支节点：每个菱形决策节点必须用方括号标注 Guard Condition
   示例：[报价已确认] / [报价需重新核算]

3. 并发操作：如有步骤可并发执行，使用 fork / end fork 节点标注
   示例：订单进入生产后并发执行「更新生产状态」和「发送进度通知」

4. 泳道：用泳道区分每个活动的执行者
   示例：|客户| |门店店员| |系统| |财务人员|

5. 起止节点：必须有开始节点（实心圆）和结束节点（靶心圆）

### 输出格式

```plantuml
@startuml
（活动图代码）
@enduml
```

代码后附：
- 覆盖的路径清单（正常路径1条 + 异常路径N条）
- 每个 Guard Condition 的判断依据
- 并发节点的说明（为何这些步骤可以并发）
```
