# n8n 工作流1（需求开发全流程）· 提示词

> 项目：图文快印门店连锁管理系统
> 目标：生成可导入 n8n 的工作流 JSON，实现 A1→A2→A3→A4→A5→CCB→A6 的低复杂度串联。

## 系统角色设定

你是一名 n8n 工作流架构师，负责把需求工程全流程设计为可执行的 n8n 工作流。

当前项目使用本地 FastAPI + CrewAI 应用 `agent-app`，默认地址为：

```text
http://127.0.0.1:8000
```

Obsidian 知识库位于：

```text
obsidian-vault/
```

核心约束：

- API key 只能通过 n8n Webhook 请求体临时传递，不写入 JSON 文件、Markdown 或仓库。
- A5 只作为需求验证与 CCB 评审建议，不再自动闭环修复。
- A6 支持 CCB 带风险批准；若 A5 未通过，创建基线时必须传 `acceptA5Risks: true`。
- 不调用已删除接口 `/api/a5/auto-repair` 或 `/api/a3/repair-from-a5`。

## 工作流整体结构

工作流名称：

```text
需求开发全流程（工作流1）
```

触发方式：

```text
POST /webhook/requirements-workflow-1/start
```

启动请求体：

```json
{
  "apiKey": "YOUR_BAILIAN_KEY",
  "agentBaseUrl": "http://127.0.0.1:8000",
  "acceptA5Risks": false,
  "ccbConclusion": "通过（无保留意见）"
}
```

默认涉众：

```json
[
  { "id": "clerk", "name": "门店店员" },
  { "id": "manager", "name": "门店店长" },
  { "id": "operation", "name": "总部运营管理员" },
  { "id": "finance", "name": "财务人员" },
  { "id": "customer", "name": "客户" },
  { "id": "delivery", "name": "配送/外协人员" },
  { "id": "admin", "name": "系统管理员" }
]
```

## 节点设计

| 节点编号 | 节点名称 | n8n 节点类型 | 输入来源 | 输出目标 | 特殊说明 |
|---|---|---|---|---|---|
| 1 | 启动需求开发流程 | Webhook | 外部 POST 请求 | 准备执行参数 | 接收 `apiKey`、`agentBaseUrl`、风险确认参数 |
| 2 | 准备执行参数 | Code | Webhook body | A1b 批量访谈并保存 | 设置默认涉众、轮次数、项目名 |
| 3 | A1b 批量访谈并保存 | HTTP Request | 执行参数 | A2 需求质量分析 | 调用 `/api/n8n/a1b-batch`，由 agent-app 执行七类涉众三轮并保存 |
| 4 | A2 需求质量分析 | HTTP Request | A1 保存结果 | A3 UML 建模 | 调用 `/api/a2/analyze`，报告写入 `wiki/summaries/` |
| 5 | A3 UML 建模 | HTTP Request | 最新 notes 与 A2 报告 | A4 SRS 生成 | 调用 `/api/a3/model`，产物写入 `wiki/summaries/UML模型/` |
| 6 | A4 SRS 生成 | HTTP Request | 最新 A1/A2/A3 材料 | A5 需求验证 | 调用 `/api/a4/generate`，生成 `SRS-初稿-vX.Y.md` |
| 7 | A5 需求验证 | HTTP Request | 最新 SRS、notes、UML | CCB 审批等待 | 调用 `/api/a5/validate`，只生成评审报告 |
| 8 | CCB 审批等待 | Wait/Webhook | 人工审批回调 | 解析 CCB 决策 | 暂停等待 `/webhook/requirements-workflow-1/ccb-approval` |
| 9 | 解析 CCB 决策 | Code | CCB 回调 body | 是否创建基线 | 支持 `approved`、`approvedWithRisk`、`rejected`、`revise` |
| 10 | 是否创建基线 | IF | CCB 决策 | A6 或终止 | approved / approvedWithRisk 进入 A6 |
| 11 | A6 创建需求基线 | HTTP Request | CCB 结论和风险确认 | 完成通知 | 调用 `/api/a6/create-baseline` |
| 12 | 完成通知 | Code | A6 输出 | 结束 | 输出基线目录、文件列表、需求数量 |
| 13 | 审批未通过或要求返修 | Code | CCB rejected/revise | 结束 | 不自动循环；提示回页面手动返修 |

## CCB 审批回调

默认路径：

```text
POST /webhook/requirements-workflow-1/ccb-approval
```

说明：若 n8n Wait 节点在执行详情中提供运行时 resume URL，则 CCB 回调应优先使用该 URL，固定路径作为工作流设计中的默认回调路径。

### 审批通过

```bash
curl -X POST http://localhost:5678/webhook/requirements-workflow-1/ccb-approval \
  -H "Content-Type: application/json" \
  -d '{
    "decision": "approved",
    "comments": "CCB 审批通过，无保留意见",
    "acceptA5Risks": false
  }'
```

### 带 A5 风险批准

```bash
curl -X POST http://localhost:5678/webhook/requirements-workflow-1/ccb-approval \
  -H "Content-Type: application/json" \
  -d '{
    "decision": "approvedWithRisk",
    "comments": "接受 A5 报告中的遗留问题，带风险进入需求基线",
    "acceptA5Risks": true
  }'
```

### 审批不通过

```bash
curl -X POST http://localhost:5678/webhook/requirements-workflow-1/ccb-approval \
  -H "Content-Type: application/json" \
  -d '{
    "decision": "rejected",
    "comments": "风险不可接受，本轮终止"
  }'
```

### 要求退回 A4

```bash
curl -X POST http://localhost:5678/webhook/requirements-workflow-1/ccb-approval \
  -H "Content-Type: application/json" \
  -d '{
    "decision": "revise",
    "comments": "请退回 A4 修订 SRS 后重新运行 A5",
    "returnTo": "A4 SRS 生成"
  }'
```

## 标准化输出格式建议

A5 报告供 CCB 阅读，不再要求 n8n 自动解析为阻塞门禁。若后续需要 JSON 摘要，可使用以下格式：

```json
{
  "passed": false,
  "recommendation": "退回A4",
  "summary": {
    "blocking": 1,
    "important": 2,
    "suggestion": 3
  },
  "issues": [
    {
      "id": "ISS-A5-001",
      "reqId": "REQ-ORD-001",
      "severity": "重要",
      "description": "SRS 与原始需求存在偏差",
      "suggestion": "建议 A4 单次返修"
    }
  ],
  "ccbSummary": "作为 CCB 评审输入，由人工决定是否带风险批准"
}
```
