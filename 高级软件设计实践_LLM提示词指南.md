# 高级软件设计实践 · LLM 操作提示词指南

> **使用说明**：本文档按实训流程顺序整理，每个步骤包含【背景】【你的任务】【输入材料】【输出要求】四个部分，可直接复制粘贴发送给大模型。根据你的实际项目内容替换 `{{ }}` 中的占位符。

### STEP 0｜环境搭建 · compile.js 编译脚本

```
【背景】
我正在用 Obsidian 搭建一个四层知识库，目录结构如下：
- raw/notes/          ← 原始涉众对话记录
- wiki/summaries/     ← 分析提炼产物（需求清单、UML、SRS草稿）
- wiki/baselines/     ← 经审批的正式基线（BL-YYYYMMDD-NN/）
- archive/            ← 归档层

【你的任务】
帮我写一个 Node.js 脚本 compile.js，执行以下四类检查：
1. 结构完整性：四层目录是否存在，文件命名是否符合规范
   - raw/notes 文件命名格式：{涉众角色}-{YYYYMMDD-HHMM}-需求记录.md
   - wiki/summaries 文件命名格式：{文档类型}-v{版本号}.md
   - wiki/baselines 目录命名格式：BL-YYYYMMDD-NN
2. 链接有效性：所有 Obsidian [[双向链接]] 指向的目标文件是否存在
3. 内容格式：Markdown 表格完整性、代码块是否闭合
4. 基线一致性：基线目录中的文件是否与 wiki/summaries 源文件内容一致

【输出要求】
- 只报告问题，不自动修复
- 输出格式：每条问题包含 [类型] 文件路径: 具体问题描述
- 所有检查通过时输出：✅ compile.js 编译通过，知识库结构正常
```

### STEP 1｜涉众 AI 智能体设计（CrewAI · A1）

**每个涉众角色发送一次，替换涉众信息后使用。**

```
【背景】
我在用 CrewAI 框架为每个涉众设计专属 AI 智能体（A1），
用于在需求获取阶段与该涉众进行对话、记录需求。
项目：{{ 快递柜管理系统 / 进销存管理系统 / 你选择的项目 }}
涉众角色：{{ 快递员 / 系统管理员 / 财务人员 / 普通用户 / ... }}

【你的任务】
为上述涉众角色设计一个 CrewAI Agent，包含：
1. Role（角色）：一句话，体现该涉众的核心价值取向，不只是职务名称
2. Goal（目标）：面向涉众视角，描述他真正在意的系统结果，需包含可度量的期望
3. Backstory（背景故事）：至少包含：
   - 该涉众的典型工作场景（时间线、操作频率）
   - 使用旧系统/手工流程的核心痛点（具体事件，有数字支撑）
   - 对新系统的核心期望与担忧

【输出要求】
输出可直接运行的 Python 代码，格式如下：
```python
from crewai import Agent

stakeholder_agent = Agent(
    role="...",
    goal="...",
    backstory="...",
    verbose=True,
    allow_delegation=False
)
```
同时输出该 Agent 的配置说明文件（Markdown），
文件名格式：{涉众角色}-agent-config.md
```

### STEP 2｜需求质量分析（CrewAI · A2）

**所有涉众的 raw/notes 文件收集完毕后使用。**

```
【背景】
我已完成需求获取阶段，raw/notes/ 目录下存有以下涉众需求记录：
{{ 粘贴所有需求记录文件的内容，或说明文件清单 }}

【你的任务】
作为需求分析智能体（A2），对上述需求进行逐条质量检查，
检测以下六类问题：
1. 模糊：使用"尽量""一般情况下""尽可能"等不可度量词语
2. 不一致：两条需求对同一概念给出不同定义或约束
3. 矛盾：两条需求在逻辑上不能同时成立
4. 冲突：不同涉众对同一功能提出相互排斥的要求
5. 不准确：需求描述与业务事实不符
6. 不可行：需求在技术或资源约束下无法实现

【输出要求】
输出需求问题清单，格式如下：

| 需求ID | 涉众来源 | 问题类型 | 原文摘录 | 问题描述 | 严重程度（高/中/低） | 建议处理方式 |
|--------|--------|--------|--------|--------|---------------------|------------|

如存在问题需求，同时输出回退指令：
- 需重新获取的涉众：[涉众角色列表]
- 每个涉众需澄清的具体需求条目
```

### STEP 4｜SRS 初稿生成（CrewAI · A4）

```
【背景】
UML 建模已完成。以下是输入材料：
- 需求清单：{{ 粘贴需求清单 }}
- UML 模型：{{ 粘贴 PlantUML 代码或说明已有哪些图 }}
- 项目背景：{{ 项目名称与简介 }}

【你的任务】
作为需求文档智能体（A4），按照 IEEE 830 标准生成 SRS 初稿，
结构必须包含：

2. 总体描述（产品视角、产品功能、用户特征、约束、假设与依赖）
3. 具体需求

   - 数据需求

【输出要求】
- 文档总字数不少于 15000 字
- 每条功能需求格式：REQ-{模块}-{序号} | 需求描述 | 优先级 | 来源涉众
- 存入 wiki/summaries/SRS-初稿-v1.0.md
```

### STEP 8｜架构选型（六步工作流·第一步）

```
【背景】
需求工程阶段已完成，SRS 如下：
{{ 粘贴正式 SRS 关键章节，或说明系统主要特征 }}

【你的任务】
使用五维度评估框架对以下四种架构风格进行打分，并给出选型建议：
架构风格候选：单体架构 / 分层架构 / 事件驱动架构 / 微服务架构

五维度评估：
| 维度 | 说明 | 权重 |
|-----|-----|-----|
| 功能复杂度 | 业务逻辑复杂程度、模块数量 | 20% |
| 并发与性能 | 并发用户数、响应时间要求 | 25% |
| 可扩展性 | 未来业务增长的扩展需求 | 20% |
| 团队规模 | 开发人员数量与协作方式 | 20% |
| 运维能力 | 部署复杂度、监控要求 | 15% |

【输出要求】
1. 五维度打分表（每种架构风格按五维度各打1-5分，加权总分）
2. ADR-001（架构决策记录），格式：
   - 标题：ADR-001 架构风格选型
   - 状态：已决定
   - 背景：（项目特征描述）
   - 决策：（选择哪种架构风格）
   - 理由：（五维度评估结果与选择依据）
   - 备选方案：（落选的架构风格及原因）
   - 影响：（该决策对后续设计的约束）
3. ASD（架构风格声明）：将选型结论转化为可执行的架构约束规则
```

### STEP 9｜工程产物定义（六步工作流·第二步）

```
【背景】
已选定架构风格：{{ 分层架构/事件驱动/微服务 }}
ADR-001：{{ 粘贴 ADR-001 内容 }}
SRS 功能需求部分：{{ 粘贴功能需求 }}

【你的任务】
生成三种核心工程产物：

1. MDS（模块划分方案）：将系统拆分为职责单一的模块，
   每个模块说明：模块名称 | 职责描述 | 对外接口摘要 | 不属于本模块的职责

2. DTS（模块依赖拓扑）：用有向边枚举模块间的合法依赖关系，
   格式：A → B 表示 A 依赖 B（即 A 调用 B 的接口）
   同时列出明确禁止的依赖关系（禁止 X → Y）

3. ADR-002 至 ADR-004：记录模块拆分中的三个关键设计决策，
   格式同 ADR-001

【输出要求】
- MDS 以表格形式输出
- DTS 同时输出文字列表和 Mermaid 图（graph TD 格式）
- 对每个模块做四维度自检：
  完备性（职责是否覆盖所有需求）/正确性（划分是否合理）/
  一致性（与 ASD 是否一致）/有效性（是否可实现）
```

### STEP 10｜三层约束设计（六步工作流·第三步A）

```
【背景】
模块划分完成：
MDS：{{ 粘贴 MDS }}
DTS：{{ 粘贴 DTS }}
架构风格：{{ 粘贴 ASD }}

【你的任务】
设计 TLCD 三层约束体系：

1. C-ARCH（架构层约束）：全局规则，适用于整个系统
   示例格式：
   ARCH-001: Controller 层禁止直接操作数据库
   ARCH-002: 所有跨模块异步通信必须通过消息队列
   ARCH-003: 层级依赖方向为 Controller→Service→Repository，禁止反向穿透

2. C-MOD（模块层约束）：为每个模块定义职责边界和依赖白名单
   示例格式（每个模块一条）：
   MOD-ORDER-001: OrderService 职责为管理订单生命周期，禁止直接操作 locker 表
   MOD-ORDER-002: OrderService 仅允许依赖 OrderRepository、CellService、NotificationService

3. C-CODE（代码层约束）：编码规范细节
   示例格式：
   CODE-001: 所有日期字段使用 ISO 8601 格式（yyyy-MM-dd'T'HH:mm:ss）
   CODE-002: 所有金额字段使用 BigDecimal，禁止使用 float/double
   CODE-003: 所有 API 返回统一使用 Result<T> 包装类

【输出要求】
- 每层约束至少 5 条
- 每条约束有唯一编号
- 输出为 TLCD-约束文档.md
```

### STEP 11｜OpenAPI 接口契约（六步工作流·第三步B）

```
【背景】
系统模块：{{ 模块名称，如 OrderModule }}
该模块的核心接口（根据 SRS 功能需求和 MDS）：
{{ 列出需要定义的接口，如：用户取件、快递员投递、管理员开门 }}

【你的任务】
使用 OpenAPI 3.0.3 标准编写 YAML 契约文件，
每个接口必须包含：
- 路径（path）和 HTTP 方法
- 请求参数（路径参数、查询参数、请求体）
  - 每个字段的类型、是否必填、格式约束
- 响应体结构（成功响应 + 错误响应）
- 错误码规范（三层语义）：
  - HTTP 状态码（4xx/5xx）
  - 业务错误码（如 CELL_NOT_AVAILABLE）
  - 用户提示信息

【输出要求】
输出完整的 OpenAPI 3.0.3 YAML 文件，
包含 info、servers、paths、components/schemas 四个顶层节点，
错误码在 components/schemas/ErrorResponse 中统一定义。
文件命名：{{ 模块名 }}-api-v1.yaml
```

### STEP 13｜逆向校验（六步工作流·第五步）

```
【背景】
v1 代码已生成，需要验证代码是否遵守设计约束。
正向图谱（设计意图）：
- MDS：{{ 粘贴 }}
- DTS：{{ 粘贴 }}
- OpenAPI 契约：{{ 粘贴核心接口 }}
- 三层约束：{{ 粘贴 }}

已生成的 v1 代码结构：
{{ 粘贴代码目录树，或关键类/函数的代码片段 }}

【你的任务】
模拟 CodeGraph 逆向校验，按六级校验维度检测设计漂移：
1. 架构维度：层级依赖方向是否违反（如 Service 直接操作了数据库）
2. 模块维度：模块职责边界是否被侵犯
3. 拓扑维度：实际依赖关系是否超出 DTS 定义的白名单
4. 契约维度：接口实现是否与 OpenAPI 契约一致（路径、参数类型、响应结构）
5. 代码工程维度：是否违反代码层约束（日期格式、金额类型等）
6. 需求溯源维度：每个 REQ-XX 需求是否在代码中有对应实现

【输出要求】
输出 RCR 逆向校验报告，格式：

## RCR 逆向校验报告 v1
校验时间：{{ YYYY-MM-DD }}

### 校验结果汇总
| 校验维度 | 漂移数量 | 严重程度 |
|---------|---------|---------|

### 漂移详情
| 编号 | 维度 | 位置（文件:行号） | 违反的约束编号 | 问题描述 | 严重程度 | 修复建议 |
|-----|-----|---------------|-------------|--------|---------|--------|

### 结论
漂移总数：{{ N }}处（高/中/低：X/Y/Z）
[ ] 通过——漂移≤2处，可进入下一步
