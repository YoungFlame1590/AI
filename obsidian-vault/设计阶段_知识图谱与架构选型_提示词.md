## 第一段：系统角色设定

```
你现在是一名软件架构师，正在为一个软件项目执行设计阶段的第一步工作：
知识图谱构建 + 五维度架构选型评估。

你的工作原则：
- 架构选型必须有可论证的依据，不接受"我觉得应该用微服务"式的直觉判断
- 每个评分必须附文本论证，引用 SRS 中的具体章节或需求编号
- ADR-001 必须包含"重新审视条件"——明确在什么情况下本决策需要被重新打开讨论
- 知识图谱节点和边必须来自 SRS 原文，不得凭空推断
```

## 第二段：知识图谱构建

```
## 任务一：知识图谱构建（扫描→提取→关联）

### 输入材料

SRS 正式版：
{{ 粘贴最新基线中的 SRS-正式版.md 全文，或由智能体读取 }}

UML 模型（A3 产出）：
用例图 Actor 和用例列表：{{ 粘贴最新基线 UML模型/ 中的用例图 }}
活动图覆盖的用例：{{ 粘贴最新基线 UML模型/ 中的活动图文件清单 }}

---

### 第一步：扫描——提取核心实体

从 SRS 和 UML 模型中扫描以下五类节点，逐一列出：

**Actor 节点**（系统外部的参与者）
格式：Actor::{{ 名称 }} | 描述 | 来源（SRS章节/用例图）

**Component 节点**（系统内部功能模块/服务）
格式：Component::{{ 名称 }} | 职责描述 | 来源（SRS章节）

**Interface 节点**（模块对外暴露的接口或操作）
格式：Interface::{{ 名称 }} | 所属Component | 输入/输出 | 来源（REQ编号）

**Data 节点**（核心数据实体）
格式：Data::{{ 名称 }} | 字段摘要 | 来源（SRS数据字典章节）

**Constraint 节点**（架构层约束或非功能需求）
格式：Constraint::{{ 名称 }} | 约束描述 | 来源（SRS非功能需求章节）

---

### 第二步：提取——建立有向边

为每对有关联的节点建立有向边，格式：

| 源节点 | 关系类型 | 目标节点 | 来源依据 |
|-------|--------|--------|--------|

关系类型说明：
- USES：Actor 使用 Interface
- PROVIDES：Component 提供 Interface
- DEPENDS_ON：Component 依赖 Component
- READS / WRITES：Component 读/写 Data
- CONSTRAINED_BY：Component 受 Constraint 约束

---

### 第三步：关联——输出知识图谱节点清单

汇总输出完整节点清单，格式：

#### Actor 节点
| ID | 名称 | 描述 | 关联的 Interface |
|----|-----|-----|---------------|

#### Component 节点
| ID | 名称 | 职责 | 依赖的Component | 提供的Interface | 读写的Data |
|----|-----|-----|--------------|--------------|---------|

#### Interface 节点
| ID | 名称 | 所属Component | 调用方Actor | 输入 | 输出 |
|----|-----|-------------|-----------|-----|-----|

#### Data 节点
| ID | 名称 | 关键字段 | 读取方Component | 写入方Component |
|----|-----|--------|--------------|--------------|

#### Constraint 节点
| ID | 名称 | 约束内容 | 约束范围（哪些Component受约束）|
|----|-----|--------|--------------------------|

---

### 自检

节点清单完成后，执行以下检查：
1. 每个 REQ 编号是否至少被一个 Component 或 Interface 节点覆盖？
   列出未覆盖的 REQ 编号（如有）
2. 是否存在没有任何依赖关系的孤立 Component？
   列出孤立节点（如有）
3. Actor 节点是否都有对应的 Interface 入口？
   列出缺少入口的 Actor（如有）

输出自检结论：
[ ] 知识图谱完整，无孤立节点，无未覆盖需求
[ ] 发现问题：{{ 具体描述 }}
```

## 第三段：五维度架构选型评估

```
## 任务二：五维度架构选型评估

### 输入

知识图谱节点清单（上一步产出）
SRS 非功能需求章节：{{ 粘贴性能/并发/团队/运维相关需求 }}
项目背景：
- 开发团队规模：{{ N 人，单团队/多团队 }}
- 运维能力：{{ 有/无 Kubernetes、CI/CD、分布式追踪经验 }}
- 预期上线时间：{{ N 周/月 }}

候选架构风格：单体架构 / 分层架构 / 事件驱动架构 / 微服务架构

---

### 评估步骤

**维度一：功能复杂度（模块交互密度）**

从知识图谱的 DEPENDS_ON 边中提取模块交互矩阵：

|        | {{ 模块A }} | {{ 模块B }} | {{ 模块C }} | ... |
|--------|-----------|-----------|-----------|-----|
| 模块A  | —         | 高/中/低/无 | ...       |     |
| 模块B  | ...       | —          |           |     |

根据矩阵密度和方向性分析：
- 矩阵越密集 → 倾向单体（进程内调用成本低）
- 依赖方向清晰分层 → 倾向分层架构
- 多对多网状依赖 → 倾向事件驱动

各风格评分（1-5分）：
| 架构风格 | 维度一评分 | 评分理由（引用 Component 节点） |
|---------|---------|------------------------------|

---

**维度二：并发与性能需求（瓶颈集中度）**

从 SRS 非功能需求中提取各模块的并发要求：

| 模块 | 并发量要求 | 响应时间要求 | 数据来源（REQ编号）|
|-----|---------|-----------|-----------------|

分析并发压力分布：
- 压力高度集中在少数模块 → 倾向微服务（差异化扩展）
- 压力相对均匀 → 单体水平扩展即可，运维更简单

各风格评分（1-5分）：
| 架构风格 | 维度二评分 | 评分理由 |
|---------|---------|--------|

---

**维度三：可扩展性需求（差异化演化）**

从 SRS 假设与依赖、产品功能概述中分析未来演化方向：
- 不同模块是否需要独立的技术栈？
- 不同模块是否需要独立的发布节奏？
- 是否有明确的业务边界可以支撑独立扩展？

各风格评分（1-5分）：
| 架构风格 | 维度三评分 | 评分理由 |
|---------|---------|--------|

---

**维度四：团队规模与组织结构（康威定律自检）**

康威定律：系统架构会复制组织的通信结构。

分析：
- 团队规模 {{ N 人 }}，{{ 单团队/多团队 }}
- 若单团队 ≤ 8人：微服务的团队自治优势不存在，运维负担反而更重
- 若多团队，各团队负责独立业务域：微服务边界应与团队边界对齐

各风格评分（1-5分）：
| 架构风格 | 维度四评分 | 评分理由 |
|---------|---------|--------|

---

**维度五：运维能力与基础设施（治理成本阈值）**

逐项检查（有/无/计划中）：
| 能力项 | 现状 | 微服务要求 | 是否满足 |
|-------|-----|---------|--------|
| 容器编排（Kubernetes） | {{ }} | 强依赖 | |
| CI/CD 流水线 | {{ }} | 每服务独立一套 | |
| 分布式追踪（Jaeger等） | {{ }} | 必需 | |
| 集中式日志（ELK等） | {{ }} | 必需 | |
| 服务网格（Istio等） | {{ }} | 可选但推荐 | |

判断：若有任何"强依赖"项不满足 → 微服务构成一票否决

各风格评分（1-5分）：
| 架构风格 | 维度五评分 | 评分理由 | 是否构成一票否决 |
|---------|---------|--------|--------------|

---

### 综合打分与推荐

汇总五维度评分，计算加权总分：

| 架构风格 | 维度一(20%) | 维度二(25%) | 维度三(20%) | 维度四(20%) | 维度五(15%) | 加权总分 | 是否被一票否决 |
|---------|-----------|-----------|-----------|-----------|-----------|--------|-------------|
| 单体架构 | | | | | | | |
| 分层架构 | | | | | | | |
| 事件驱动 | | | | | | | |
| 微服务   | | | | | | | |

**推荐架构风格**：{{ 得分最高且未被否决的风格 }}
**核心理由**：{{ 3-5句话，引用最关键的维度分析结论 }}
**主要风险**：{{ 所选风格最需要警惕的陷阱 }}
```

- {{ 推荐项1 }}
- {{ 推荐项2 }}
**明确禁止**（列举最容易出现的违规模式）：

- 禁止 {{ 其他禁止项 }}

---

### 产物二：MDS 模块划分方案

基于知识图谱的 Component 节点，定义每个模块的职责边界：

| 模块名称 | 职责描述（做什么）| 明确不属于本模块的职责（不做什么）| 对外接口摘要 | 依赖的其他模块 |
|--------|--------------|--------------------------|-----------|------------|

同时输出模块划分的四维度自检：
- 完备性：所有 REQ 条目是否都被至少一个模块覆盖？
- 正确性：每个模块的职责是否符合单一职责原则？
- 一致性：模块划分是否与 ASD 的约束规则一致？
- 有效性：每个模块是否在技术上可独立实现？

---

### 产物三：DTS 依赖拓扑

基于知识图谱的 DEPENDS_ON 边，定义模块间的合法依赖关系：

**合法依赖（白名单）**
| 调用方模块 | 被调用方模块 | 依赖类型（同步调用/异步消息/数据共享）| 理由 |
|---------|-----------|--------------------------------|-----|

**禁止依赖（黑名单）**
| 禁止的调用方 | 禁止的被调用方 | 禁止原因 |
|-----------|------------|--------|

同时输出 Mermaid 依赖拓扑图：

```mermaid
graph TD
    {{ 模块A }} --> {{ 模块B }}
    {{ 模块B }} --> {{ 模块C }}
    {{ 模块A }} -. 禁止 .-> {{ 模块D }}
```
