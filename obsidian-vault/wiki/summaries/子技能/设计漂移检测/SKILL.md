# 技能名称：设计漂移检测

## 操作定义

比对设计资产与实际代码，检测架构职责、依赖拓扑、接口契约、需求与角色覆盖四类设计漂移，并输出 RCR 风格报告。

## 输入参数

| 参数名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| mdsFile | file | 是 | 模块划分方案 MDS |
| dtsFile | file | 是 | 依赖拓扑 DTS |
| tlcdFile | file | 是 | 三层约束文档 TLCD |
| openapiFile | file | 是 | OpenAPI 契约 |
| codeRoot | directory | 是 | 待校验代码根目录 |
| outputPath | string | 是 | RCR 报告输出路径 |

## 执行步骤

1. 读取 MDS，提取模块名称、职责边界和不属于本模块的职责。
2. 读取 DTS/TLCD，提取合法依赖、禁止依赖和层级方向约束。
3. 扫描代码包结构、Controller 路径、Service 调用、Repository 使用和前端入口。
4. 比对 OpenAPI paths 与 Controller 实现，标记缺失、额外或参数不一致的接口。
5. 检查七类角色、关键需求模块和变更请求能力是否在代码中有实现痕迹。
6. 生成漂移详情、严重程度和是否通过的结论。

## 输出格式

```markdown
# RCR 逆向校验报告

## 校验结果汇总

| 漂移类型 | 数量 | 高 | 中 | 低 | 结论 |
|---|---:|---:|---:|---:|---|

## 漂移详情

| 编号 | 类型 | 位置 | 违反约束 | 问题描述 | 严重程度 | 修复建议 |
|---|---|---|---|---|---|---|

## 结论

- 高严重度漂移：0
- 是否通过：是/否
```

## 验证步骤

- [ ] 至少覆盖架构职责、依赖拓扑、接口契约、需求与角色覆盖四类漂移。
- [ ] 高严重度漂移必须给出可定位的文件或接口路径。
- [ ] Workbench 或演示扩展接口需与核心契约接口分开标记。
- [ ] 报告结论包含通过/不通过。
- [ ] 输出不包含 API key、令牌或本机私密路径。
