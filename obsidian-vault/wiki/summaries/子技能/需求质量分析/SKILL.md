# 技能名称：需求质量分析

## 操作定义

对涉众原始记录或需求清单执行需求质量检查，识别模糊、不一致、矛盾和冲突问题，并输出可回退到 A1/A2 的结构化问题清单。

## 输入参数

| 参数名 | 类型 | 必填 | 说明 |
|---|---|---|---|
| projectName | string | 是 | 项目名称，默认为图文快印门店连锁管理系统 |
| sourceFiles | file[] | 是 | `raw/notes/` 或需求清单 Markdown 文件 |
| stakeholderScope | string[] | 否 | 需要重点检查的涉众角色 |
| severityPolicy | object | 否 | 高/中/低严重程度判定规则 |
| outputPath | string | 是 | 输出问题清单的目标路径 |

## 执行步骤

1. 读取输入文件，按涉众、文件名、原文摘录建立需求候选条目。
2. 为每条条目检查模糊词、不可量化指标、缺少触发条件和缺少验收标准的问题。
3. 横向比较不同涉众对同一对象、流程、权限、金额、时限和状态的描述。
4. 将问题归类为模糊、不一致、矛盾、冲突，并按影响范围判定严重程度。
5. 对高严重度问题生成回退建议，明确需回访涉众和追问话术。
6. 生成 Markdown 问题清单，保存到 `outputPath`。

## 输出格式

```markdown
# 需求质量分析结果

| 问题ID | 涉众来源 | 问题类型 | 严重程度 | 原文摘录 | 问题描述 | 建议处理 |
|---|---|---|---|---|---|---|
| RQ-001 | 门店店员 | 模糊 | 中 | ... | ... | ... |

## 回退建议

| 涉众 | 追问目标 | 建议话术 | 期望补充信息 |
|---|---|---|---|
```

## 验证步骤

- [ ] 每条问题都有问题ID、涉众来源、问题类型和严重程度。
- [ ] 问题类型只使用模糊、不一致、矛盾、冲突四类。
- [ ] 高严重度问题必须包含回退建议。
- [ ] 输出不包含 API key、令牌或本机私密路径。
- [ ] 输出文件可被 `compile.js` 检查通过。
